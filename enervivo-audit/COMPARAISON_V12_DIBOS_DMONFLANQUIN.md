## Légende verdicts

- ✅ **EXACT** : le fichier exact attendu par V12 est dans `found_files` de l'audit.
- ⚠️ **AUTRE** : le doc est `present`/`ambiguous` mais le fichier classé n'est pas celui attendu par V12 (faux positif probable, ou multi-candidats acceptables).
- ❌ **missing** : aucun fichier classé sur ce doc.
- ❌ **not_applicable** : règle `Cas par cas` (V12) ou MSA-vs-AgriPV a écarté la ligne — la vérité-terrain dit pourtant que le fichier existe.
- ❓ **doc absent du rapport** : le doc V12 n'a pas de ligne correspondante dans le rapport audit (typiquement nouveau doc V12 type `DT/DICT résumé` non encore propagé au référentiel JSON).

## Méthodologie

1. Extraction des colonnes `Lien_DIBOS_H` et `Lien_DMONFLANQUIN` du V12 → pour chaque doc du référentiel (107 lignes), 0 ou 1 lien = chemin du fichier exact attendu.
2. Dernier audit `completed` par projet (table `audits`).
3. JSON `result` aplati ; matching V12↔audit par `(jalon, nom de document)` normalisé.
4. Verdict : EXACT si basename du `Lien_*` V12 figure dans `found_files[].file_name` (case-insensitive) ; AUTRE si found_files non vide mais aucun match ; MISS/NA si pas de found_files ; ABS si la ligne audit n'existe pas.

## Limites / hypothèses

- Le matching multi-instances (PV huissier 1/2/3) se fait par **ordre d'apparition**. Si l'audit a fusionné les 3 PV en une seule ligne, les instances 2 et 3 apparaîtront artificiellement en ❓ ou ❌.
- Le verdict regarde **uniquement le basename** du `Lien_*` V12. Un fichier renommé identique mais à un chemin différent reste ✅ ; deux homonymes dans dossiers différents indistinguables.
- L'écart entre `total_expected` rapport (91/91) et les 46/30 attendus V12 vient du fait que V12 ne renseigne `Lien_*` que pour les docs **réellement présents** côté projet.
- Patterns d'erreur connus (cf SESSION.md) : `.xlsb` TADD anciens souvent ratés, JPG identité (CNI/livret/RIB) ratés via vision LLM, multi-instances mal gérées, versioning plan de masse / TADD / qualif par jalon confondus.

## Contre-argument

Le score `EXACT` sous-estime la qualité réelle : un fichier classé sous le **bon doc V12** mais avec un **autre nom** (versioning, copie SharePoint) est marqué ⚠️ AUTRE, alors que le besoin métier est satisfait. Inversement, ⚠️ peut masquer un vrai faux positif (audit a classé un fichier sans rapport sous ce doc). Trancher exigerait revue humaine fichier par fichier.

Métrique plus charitable : **EXACT + AUTRE** = audit a livré un fichier `present` sur cette ligne — DIBOS_H 29/46 (63%), DMONFLANQUIN 24/30 (80%).
